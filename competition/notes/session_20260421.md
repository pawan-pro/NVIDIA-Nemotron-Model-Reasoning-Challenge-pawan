# Session Note — 2026-04-21

## Objective

Establish the **first valid end-to-end baseline submission pipeline** for the Kaggle **NVIDIA Nemotron Model Reasoning Challenge**, before starting prompt, synthetic-data, or LoRA research experiments.

## Why this was the right first session

The highest-risk blocker in this competition is **submission validity**, not model quality.
The competition requires a **LoRA adapter** for **Nemotron-3-Nano-30B**, packaged as **`submission.zip`**, with **`adapter_config.json`** included. The evaluator loads the base model with the submitted adapter and scores **accuracy**, while prioritizing final answers extracted from `\boxed{}`.  

So Session 1 was treated as a **pipeline-validation session**, not a leaderboard-optimization session.

## Session learnings

1. **Submission validity comes first**
   A weak but valid baseline submission is more valuable than an unsubmitted better idea.

2. **Do not over-customize the baseline on Day 1**
   The submission demo path is the right place to start.

3. **Packaging must be treated as part of the model pipeline**
   Zip contents matter.

4. **This benchmark likely rewards output discipline**
   The metric prioritizes boxed final answers, so prompt-format experiments should be the first real ablation after the baseline is confirmed.

## Recommended next step after submission result

Once the submission result is known, the next experiment should be:

### First real experiment: prompt-only formatting ablation

**Hypothesis**
A stricter final-answer format can improve measured accuracy because the evaluator explicitly prioritizes answers in `\boxed{}`. 

**Why it may help on this benchmark**
This task includes short, exact answers across binary, symbolic, and text outputs; formatting mistakes or answer drift may cost accuracy even when reasoning is close.

**Implementation steps**

* Freeze the baseline packaging notebook
* create one prompt-only experiment notebook
* compare:

  * baseline prompt
  * stricter “single final boxed answer only” prompt
  * stricter “verify rule, then output only `\boxed{}`” prompt

**Risks / failure modes**

* too much compression may reduce reasoning quality
* excessive instruction may cause style overfit

**How to evaluate success**

* run on a fixed local train slice
* label errors as:

  * formatting error
  * rule error
  * arithmetic slip
  * multiple-candidate / answer-drift failure
